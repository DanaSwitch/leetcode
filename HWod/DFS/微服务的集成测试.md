# 微服务的集成测试

## 思路

很多时候觉得代码写起来“很乱”，是因为我们**混淆了“思考过程”和“代码实现”**。当你面对这种问题时，可以按照下面这个“三步走”的思维框架来拆解，这样思路会非常清晰。

------

### 第一步：抽象出问题的“本质” (画图)

不要直接写代码，先在纸上画图。把服务看作圆圈，把依赖看作连线。

- **提问**：每个服务要启动，必须先等谁？
- **转化**：既然 $i$ 依赖 $j$，那 $i$ 的完成时间一定等于：**$j$ 的完成时间 + 我自己跑的时间**。
- **发现**：这就是一个**递归定义**。如果 $i$ 没有任何前置依赖，它的时间就是 `self_cost`。如果它有多个依赖，它必须等到**最慢的那个依赖项**跑完。

------

### 第二步：逻辑自顶向下设计 (伪代码)

在写具体代码前，先用自然语言描述逻辑，这叫“自顶向下”。

```Plaintext
计算服务 i 的时间：
1. 先看看有没有算过？（有的话，直接拿来用 -> 记忆化）
2. 还没算过的话：
   a. 获取我自己要跑多久（self_cost）。
   b. 找出我依赖的所有人（j1, j2...）。
   c. 如果没依赖，我就直接跑。
   d. 如果有依赖，我要等他们都跑完，所以我需要 max(j1时间, j2时间...)。
   e. 我的总时间 = self_cost + max(依赖时间)。
3. 把算出来的总时间存进记账本(memo)。
```

**这一步能保证你思路不乱，因为你已经把核心逻辑抠出来了。**

------

### 第三步：映射到代码结构 (填空)

有了上面的逻辑，翻译成 Python 就像“填空”一样简单：

1. **准备环境**：你需要 `use_time` 矩阵（数据源）、`memo`（记账本）。
2. **写递归函数**：
   - 函数签名：`def get_ready_time(i, ...)`。
   - 开头：`if i in memo: return memo[i]` （这就是刚才说的“先看看有没有算过”）。
   - 结尾：`memo[i] = result` （这就是“算完记账”）。
3. **处理输入**：主函数负责把 `input` 变成程序能懂的 `use_time` 矩阵和 `target_k`，最后调用递归函数。

------

### 为什么你会觉得“乱”？

通常是因为你试图**在大脑里同时模拟整个递归过程**。其实你根本不需要这样做！

**递归的思维诀窍在于：**

在写 `get_ready_time(i)` 时，你要**假设** `get_ready_time(j)` 已经帮你把子任务计算好了，你只需要直接拿结果去用，而不需要关心 `j` 到底是怎么算出来的。

**这种“只关注当前这一步，信任子递归”的思维，是攻克所有递归难题的终极武器。**





## 错误更正

### 主要问题点：

1. **列表推导式语法错误**： 你的代码：`[j for j in range(n) and use_time[i][j] == 1 and i != j]`
   - 在 Python 的列表推导式中，`if` 关键字必须写在 `for` 后面。现在的写法在逻辑上会被解释为对 `range(n)` 的布尔值判断，这会直接报错。
   - **修正**：`[j for j in range(n) if i != j and use_time[i][j] == 1]`
2. **数据类型问题**： 你的代码：`martix = sys.stdin.readline().split()`
   - `split()` 出来的结果是**字符串列表**（例如 `['1', '0', '10']`），而矩阵中的数字应该是整数。直接使用 `use_time[i][j] == 1` 会因为字符串与整数比较而失效。
   - **修正**：`row = [int(x) for x in sys.stdin.readline().split()]`
3. **函数返回值缺失**： 你的代码：`print(solve())`
   - 你的 `solve()` 函数内部没有写 `return`，所以 `print(solve())` 实际上会打印出 `None`。你应该在 `solve` 函数结尾返回 `least_time(...)` 的结果。
